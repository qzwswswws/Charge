# P5 多随机种子复现与统计检验计划

## 1. 文档目的

本文档用于将 `P5` 从“概念性待办”收束为一套可执行的统计复现方案。  
本轮计划的核心目标不是继续发散新模型，而是提高当前论文主结果的**统计可信度**。

换句话说，`P5` 的主要作用是把现有结论从：

- “单次运行结果成立”

推进到：

- “在多随机种子下结果仍然稳定，并且关键差异具备统计上的可信性”

---

## 2. 为什么 P5 现在很重要

截至当前，论文已经具备以下主线证据：

- `22导 / 4分类` baseline
- `22 -> 8 -> 3 -> 2` 四分类退化
- `C3/C4 / 2分类` 工程落地
- `KD` 补充性补偿实验
- `P4` 通道组合 pilot

但这些结果大多仍是：

- `seed = 42`
- 单次运行

因此，P5 的价值非常高，因为它能回答三类评审最常问的问题：

1. 这些结果是否只是偶然跑得好？
2. `2导 / 2分类` 的优势是否在多随机种子下仍然成立？
3. `2导 / 4分类` 与 `2导 / 2分类` 的差异到底有多稳定？

---

## 3. 当前执行约束

当前机器存在一个现实约束：

- 本机 CUDA 栈当前报 `Driver/library version mismatch`
- `torch.cuda.is_available()` 结果异常

这意味着：

- 目前最适合优先推进的是**低通道、CPU 也能接受的设置**
- 全通道 `22导 / 4分类` 多随机种子复现虽然仍可做，但不应放在最先开跑的位置

因此，`P5` 建议拆为：

- `P5-A`：立即执行的低通道统计复现
- `P5-B`：在驱动修复后再推进的全通道统计复现

---

## 4. 推荐的总体策略

## 4.1 核心原则

`P5` 不要一上来铺满所有设置，而应按“论文价值 / 耗时 / 与主线的贴合度”分层推进。

建议遵循：

1. 先验证最关键的主线结论
2. 再扩到补充性结果
3. 最后再做全通道统计封口

## 4.2 推荐随机种子

第一轮建议直接使用 `3` 个种子：

- `42`
- `3407`
- `20260329`

原因：

- `3` 个种子已经能给出 `mean ± std`
- 计算量仍然可控
- 若结果方差明显过大，再决定是否扩到 `5` 个种子

---

## 5. P5 的分阶段安排

## 5.1 P5-A1：`C3/C4 / 2-class` 多种子复现

### 推荐度

`最高`

### 原因

这是当前论文最核心、也最工程化的一条结果：

- 它直接支撑“极低通道场景下，左右手二分类可行”
- 它比 `KD` 更像主线
- 它比全通道统计更贴近论文的低通道贡献点

### 推荐设置

- 配置：`c3c4`
- 类别：`1,2`
- 被试：`1~9`
- 种子：`3` 个
- 脚本入口：[conformer_degradation.py](/home/woqiu/下载/git/MI_Algorithm_Workbench/baselines/conformer_degradation.py)

### 推荐输出

- `multi_seed_c3c4_2class_raw.csv`
- `multi_seed_c3c4_2class_summary.csv`
- 每被试 `mean ± std`
- 全被试平均 `mean ± std`

### 预估耗时

- 首批结果：`2~4 小时`
- 完整收尾：`半天以内`

---

## 5.2 P5-A2：`C3/C4 / 4-class` 多种子复现

### 推荐度

`很高`

### 原因

这条结果与 `P5-A1` 一起，构成论文中最关键的一组统计对照：

- 在同样双导条件下
- `4分类` 与 `2分类`
- 哪个更稳定、方差更小、均值差异是否稳固

### 推荐设置

- 配置：`c3c4`
- 类别：`1,2,3,4`
- 被试：`1~9`
- 种子：`3` 个

### 推荐输出

- `multi_seed_c3c4_4class_raw.csv`
- `multi_seed_c3c4_4class_summary.csv`
- 与 `P5-A1` 的 paired 对比表

### 预估耗时

- 首批结果：`2~4 小时`
- 完整收尾：`半天以内`

---

## 5.3 P5-A3：`C3/Cz/C4 / 2-class` 多种子复现

### 推荐度

`高`

### 原因

`P4` 已经说明：

- `C3/Cz/C4` 是当前 `3导` 条件下最强组合

如果我们想把 `P4` 从 pilot 提升为更像正式证据，那么最值得先扩种子、而不是先扩更多组合的，就是这条。

### 推荐设置

- 配置：`c3czc4`
- 类别：`1,2`
- 被试：先 `S1/S3/S5/S8/S9`
- 种子：`3` 个

### 为什么先不直接 9 被试

因为：

- 这条线属于 `P4` 的延伸增强，不是当前论文主轴的绝对中心
- 先用 `5` 被试看方差量级更省时间

如果表现稳定，再扩到 `9` 被试。

---

## 5.4 P5-B1：`22导 / 4-class` 多种子复现

### 推荐度

`中高`

### 原因

这条线对论文的“基线严谨性”当然重要，但考虑到当前设备状态与耗时，它不应抢在低通道主线前面。

### 推荐前提

优先等待：

- CUDA 驱动恢复正常

或者：

- 另找一台 GPU 正常的机器

### 推荐设置

- 配置：`full`
- 类别：`1,2,3,4`
- 被试：`1~9`
- 种子：`3` 个

### 风险

这条线很容易把大量时间消耗在“补基线统计口径”上，而不是继续增强论文低通道贡献。

因此建议把它放到：

- `P5-A` 跑完之后
- 且确认还有算力预算时

---

## 5.5 P5-B2：KD 多种子复现

### 推荐度

`中`

### 原因

KD 本身在论文中的定位是：

- 补充实验
- 补偿实验

因此没必要把它放在 `P5` 的最前面。

### 推荐策略

如果 `P5-A1/A2` 已经完成且时间允许，可以做：

- `KD / c3c4 / 4class`
- `5` 被试
- `3` 个种子

这样足以支撑“KD 是否只是 seed 偶然”的问题，但不会抢主线资源。

---

## 6. 推荐统计输出形式

## 6.1 必做输出

每个配置至少要有：

- 每被试 `best_acc mean ± std`
- 每被试 `aver_acc mean ± std`
- 全被试平均 `mean ± std`

## 6.2 推荐附加输出

对关键比较对：

- `c3c4 / 2class` vs `c3c4 / 4class`

建议补：

- paired difference 表
- Wilcoxon signed-rank test
- 效应方向描述

如果样本量过小或分布不稳，也可以退一步只写：

- “在 `9` 个被试中的多数被试上均呈现同方向变化”

---

## 7. 最推荐的执行顺序

若从“最快强化论文”角度出发，建议按下面顺序：

1. `P5-A1`：`C3/C4 / 2-class / 9被试 / 3 seeds`
2. `P5-A2`：`C3/C4 / 4-class / 9被试 / 3 seeds`
3. `P5-A3`：`C3/Cz/C4 / 2-class / 5被试 / 3 seeds`
4. `P5-B1`：`22导 / 4-class / 9被试 / 3 seeds`
5. `P5-B2`：`KD / 5被试 / 3 seeds`

---

## 8. 为什么这个顺序最合适

因为它优先强化的是你论文里最重要的三句话：

1. `2导 / 2分类` 是稳定可行的
2. `2导 / 4分类` 的退化不是 seed 偶然
3. 若允许增加到 `3导`，`C3/Cz/C4` 确实更强

这三条一旦有了 `mean ± std` 支撑，论文说服力会明显上一个台阶。

---

## 9. 当前建议

如果现在就准备继续动手，我最建议的不是先碰 `22导 / 4分类`，而是：

- 先启动 `P5-A1`
- 再紧接着做 `P5-A2`

原因很简单：

- 这两条最直接服务论文主线
- 当前脚本已支持 CPU 回退
- 设备异常不会完全阻断推进

一句话收束：

> `P5` 最该先做的是把双导主线从“单次结果成立”升级为“多随机种子下仍稳定成立”。
