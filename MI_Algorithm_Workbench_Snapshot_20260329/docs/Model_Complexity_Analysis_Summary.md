本轮分析的目标不是给出最终边缘部署结论，而是先建立一组**离线、可复现、可比较**的复杂度指标，包括：

## 3. 测量方法与注意事项

### 3.1 参数量

通过 `sum(p.numel())` 直接统计总参数量与可训练参数量。

### 3.2 MACs / FLOPs

使用 `THOP` 进行近似统计。

需要特别说明：

- `THOP` 原生更接近 `MACs` 统计。
- 本文中 `FLOPs` 采用 `2 × MACs` 的常见乘加换算口径。
- 由于当前模型中存在 `einsum`、`softmax` 和自定义局部注意力，`THOP` 对这些算子的覆盖可能不完全。
- 因此，这里的 `MACs/FLOPs` 更适合作为**同系列模型之间的比较指标**，而不是绝对精确的硬件级算力消耗。

### 3.3 模型大小

通过序列化 `state_dict` 并测量文件大小获得。

### 3.4 CPU 推理时间

使用本机 CPU、`batch size = 1`、固定输入长度 `1000` 点进行测试。  
该结果只能视为**离线相对比较指标**，不能替代：

- RK3568
- NPU
- RKNN 量化后延迟

的真实部署测试。

## 5. 核心结论

## 5.1 通道数下降会显著降低计算量，但对参数量影响有限

从 `22导 / 4分类 baseline` 到 `2导 / 4分类 baseline`：

- 参数量仅下降约 `5.03%`
- 模型大小仅下降约 `4.91%`
- 但 MACs 下降约 `81.96%`
- CPU 推理时间下降约 `51.86%`

这说明：

- 在当前 Conformer 架构中，**降通道对计算量和推理时间的影响远大于对参数量的影响**
- 这是因为通道数变化主要改变了前端卷积输入规模与后续 token 处理成本，而模型主体的大量线性层参数仍然保留

这条结论非常适合写进论文，因为它能解释：

- 为什么低通道不仅是硬件约束问题，也确实带来推理负担下降
- 为什么“低通道有利于部署”并不是空话

---

## 5.2 `2导 / 2分类` 与 `2导 / 4分类` 在部署复杂度上几乎相同

`baseline_c3c4_4class` 与 `baseline_c3c4_2class` 的复杂度几乎一致：

- 参数量相同量级：`0.604M`
- MACs 相同量级：`0.011G`
- 模型大小相同量级：`2.364MB`
- CPU 延迟也仅有轻微差异

这说明：

- 从部署角度看，`2导 / 2分类` 的优势并不来自“模型更小”
- 它的真正优势来自：
  - 任务本身更符合双导信息上限
  - 分类准确率明显高于双导四分类

因此，后续论文应当把：

- `2导 / 2分类` 的价值

表述为：

- **信息与任务匹配更好**

而不是：

- **模型复杂度显著更低**

---

## 5.3 Batch 1 / Batch 2 虽然更轻，但并没有换来更好准确率

相对 `baseline_c3c4_2class`：

### Batch 1

- 参数量减少约 `78.95%`
- 模型大小减少约 `77.00%`
- CPU 延迟降低约 `6.22%`

### Batch 2

- 参数量减少约 `78.69%`
- 模型大小减少约 `76.64%`
- CPU 延迟降低约 `2.50%`

但从已有准确率结果看：

- Batch 1 未超过 baseline
- Batch 2 也未超过 baseline

这意味着：

- 轻量化改造本身是有效的，确实能显著缩小模型
- 但这种轻量化没有自然转化为更高的识别性能

这条结果对论文很有价值，因为它给出一个比较成熟的工程判断：

> 在当前任务与数据规模下，单纯继续沿着“更轻”方向做结构裁剪，并不一定能换来更好的低通道准确率。

---

## 5.4 KD student 在部署端不增加推理复杂度

`kd_student_c3c4_4class` 的复杂度与 `baseline_c3c4_4class` 基本一致：

- 参数量一致
- MACs 一致
- 模型大小一致
- 推理延迟同量级

这是因为：

- KD 的 teacher 只参与训练
- 部署时只保留 student

因此可以形成一个非常适合写进论文的结论：

> 知识蒸馏的额外成本主要发生在训练阶段，而不是部署阶段；因此，只要 student 架构不变，KD 不会破坏双导模型的部署可行性。

这条结论尤其适合服务于：

- 将 KD 作为补充实验保留下来
- 同时不削弱论文对边缘部署可行性的叙述

## 7. 当前还不能过度写强的地方

虽然 `P1` 已经补上了重要证据，但下面这些表述仍然不能直接写得过强：

- 不能把本机 CPU 延迟写成 `RK3568` 延迟
- 不能把当前 THOP 近似值写成严格精确的硬件级 FLOPs
- 不能把轻量化结果直接外推为 `RKNN 量化后必然达标`

因此，当前最稳的写法是：

- 先把本轮结果作为**离线复杂度证据**
- 之后若补上真实 RK3568 / RKNN 测试，再把部署结论写强

---

## 8. 一句话总结

`P1` 的结果已经证明：  

> 在当前 Conformer 主线上，通道裁剪能显著降低计算量和推理时间；`2导 / 2分类` 的优势主要来自任务匹配而非复杂度进一步下降；而 Batch 1 / Batch 2 虽能显著减小模型，却未带来更高准确率，KD 则不会增加部署端 student 的推理复杂度。
